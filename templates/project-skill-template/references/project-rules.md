# Project Rules

## Evidence Order

1. Latest user instruction in the current thread.
2. Current local evidence: files, logs, config, generated outputs, exact errors, and current repo state.
3. Project docs and generated artifacts.
4. This skill and its references.
5. External public sources, prioritized by `sources.md`.

Historical notes are hints, not proof. Version-sensitive, dependency-sensitive, OS-sensitive, and security-sensitive facts must be rechecked.

## Communication Preferences

- Keep answers structured and cautious.
- Mark assumptions clearly when information is incomplete.
- Summarize tool/test results rather than asking the user to inspect them again.

## Project-Specific Workflow

- Add project-specific workflow rules here.
- Keep global debugging and research process in global skills.
- Avoid private credentials, account details, and noisy logs.
