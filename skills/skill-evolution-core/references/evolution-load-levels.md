# Evolution Load Levels

Use this reference whenever the configured evolution shortcut starts a skill-system update. The goal is to make the evolution action proportional to the problem.

## Default Rule

Start with the smallest level that can safely solve the issue. Escalate only when the evidence shows wider impact.

## Level 1: Light Evolution

Use for one clear lesson that maps to one target skill.

Typical signs:

- One specific mistake or near-miss.
- One obvious target skill.
- No change to routing ownership.
- No skill split, merge, absorption, devolution, or architecture change.
- The rule can be expressed as a short failure shield or workflow guardrail.

Do:

- Review the immediate context only.
- Identify the root cause in plain language.
- Patch the one target skill or one narrow reference.
- Run a minimal text check for the new rule and report the classification.

Do not:

- Read every router or reference file.
- Audit unrelated skills.
- Rewrite routing matrices.
- Run full structural validation unless the structure changed.
- Turn the task into a broad cleanup.

## Level 2: Standard Evolution

Use when the lesson affects routing, classification, or more than one closely related skill.

Typical signs:

- The target skill is not obvious.
- The rule may belong in global guidance, a router, or a project skill.
- Two or three files need coordinated edits.
- The change updates trigger wording, routing categories, or quality gates.
- There is meaningful risk of duplicate or over-narrow rules.

Do:

- Read the router or references needed to classify the destination.
- Patch the minimal set of affected files.
- Update any required public-safe ledgers or templates.
- Run targeted checks for trigger wording, duplicates, privacy, and mojibake.

## Level 3: Full Evolution

Use only for evolution-system architecture work.

Typical signs:

- Creating, splitting, merging, absorbing, or degrading skills.
- Changing `skill-evolution-core`, `skill-evolution-router`, major routing matrices, or capability registries.
- Installing or absorbing an external skill, plugin, tool, or app.
- A devolution pass over bloated, stale, conflicting, or progress-blocking rules.
- Multiple skills need coordinated ownership changes.

Do:

- Run the full core workflow.
- Apply lifecycle and devolution rules when relevant.
- Validate structure when new skills, metadata, scripts, or generated routing support changed.
- Report changed files, classification, validation, and remaining risk.

## Regression Example

Case: one successful artifact or result is visible, but the agent continues into a lower-quality fallback because no durable file path is visible.

Expected classification:

- Light Evolution

Expected action:

- Patch only the relevant artifact or output-preservation skill with a failure shield.
- Do not run a full router audit or multi-skill review.

Expected conclusion:

- The durable rule is a generic preservation guardrail, not a project-specific rule.
